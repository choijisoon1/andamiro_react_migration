### 수정 1 — 초대 미리보기 공개 정보 제한

**수정일** 2026-07-24

**파일** `supabase/functions/get-exchange-invitation-preview/index.ts`

**관련 커밋** `2a6096e` (`보안: 초대 미리보기 공개 정보 제한`)

기존 함수는 초대 토큰을 받은 뒤 서비스 역할로 `exchange_posts`를 조회하면서
`content`, `image_url`, `user_id`까지 반환했다. 화면에서는 제목만 표시했지만,
비밀번호를 입력하기 전에 API 응답을 직접 확인하면 공유일기 본문과 이미지
주소를 볼 수 있었다. 이 동작은 최초 Vue 원본에도 존재했다.

다음과 같이 수정했다.

- 함수 내부에서 `Authorization` Bearer 토큰을 확인한다.
- Supabase `getUser()`로 실제 로그인 사용자인지 다시 검증한다.
- 잘못된 JSON과 비어 있는 초대 토큰을 구분해 거부한다.
- 비밀번호 검증 전 미리보기 응답은 `post.id`, `post.title`만 조회한다.
- 화면에서 사용하지 않는 `content`, `image_url`, `user_id`는 응답에서 제외한다.

로컬 `npm run lint:check`는 통과했다. 실제 효력은 수정된
`get-exchange-invitation-preview` Edge Function을 Supabase에 다시 배포한 뒤
확인해야 한다. 배포 후에는 비로그인 요청이 401로 거부되는지, 정상 초대
미리보기 응답에 본문과 이미지 주소가 없는지, 올바른 비밀번호 입력 후에만
상세 화면을 읽을 수 있는지 통합 테스트한다.

### 수정 2 — 외부 워크플로를 명시적 설정으로 전환

**수정일** 2026-07-24

**파일** `supabase/functions/create-exchange-room/index.ts`

**관련 커밋** `d6327c6` (`보안: 외부 워크플로를 명시적 설정으로 변경`)

기존 함수는 `ROOM_CREATED_WORKFLOW_URL` 환경변수가 없으면 소스에 하드코딩된
ngrok 주소를 사용했다. 그 결과 운영자가 외부 자동화를 설정하지 않아도 방
ID, 작성자 사용자 ID, 제목과 생성 시각이 해당 외부 주소로 전송될 수 있었다.
이 동작도 최초 Vue 원본에 포함된 Edge Function에서 승계됐다.

다음과 같이 수정했다.

- 소스에 있던 ngrok 기본 주소를 제거했다.
- `ROOM_CREATED_WORKFLOW_URL` 값을 명시한 경우에만 외부 자동화를 활성화한다.
- 환경변수가 없으면 `exchange_room_events` 이벤트와 외부 HTTP 요청을 만들지 않는다.
- 외부 자동화를 사용하지 않아도 공유일기 방과 초대 링크 생성은 정상 완료한다.
- 이벤트 조회 필드를 실제로 사용하는 `id`로 제한한다.

로컬 `npm run lint:check`는 통과했다. 실제 효력은 수정된
`create-exchange-room` Edge Function을 Supabase에 다시 배포한 뒤 확인해야
한다. 외부 자동화를 사용하지 않는 환경에서는 `ROOM_CREATED_WORKFLOW_URL`을
설정하지 않은 채 방 생성이 성공하고 외부 요청이 발생하지 않는지 검사한다.
자동화를 사용하는 환경에서만 승인된 HTTPS 주소를 Supabase Function secret으로
설정하고 이벤트 처리 결과를 별도로 확인한다.

### 수정 3 — 개인 일기 날짜를 현지 달력 기준으로 통일

**수정일** 2026-07-24

**파일**

- `src/lib/date.js`
- `src/api/diaryApi.js`
- `src/views/main/MainView.jsx`
- `src/views/chat/ResultView.jsx`
- `src/views/advice/AdviceView.jsx`

기존 Vue 원본과 React 이관 코드 모두 `new Date().toISOString()`의 UTC 날짜를
개인 일기 `record_date`와 오늘 조언 조회에 사용했다. 한국 시간은 UTC보다
9시간 빠르므로 오전 0시부터 8시 59분 사이에 오늘 기록이 전날 날짜로 저장되거나
조언 화면이 전날 기록을 조회할 수 있었다.

다음과 같이 수정했다.

- 현지 연·월·일을 `YYYY-MM-DD`로 만드는 `formatLocalDate()` 공통 함수를 추가했다.
- 결과 화면의 오늘 날짜와 저장 API의 기본 날짜를 공통 함수로 변경했다.
- 조언 화면의 오늘 기록 조회와 메인 달력의 오늘 판정도 같은 함수를 사용한다.
- `created_at`, 초대 수락 시각처럼 실제 시점을 나타내는 서버 UTC 값은 변경하지 않았다.

`Asia/Seoul` 환경에서 `2026-07-24 00:30`을 넣어 확인했을 때 기존 UTC 방식은
`2026-07-23`, 새 `formatLocalDate()`는 `2026-07-24`를 반환했다.
`npm run lint:check`와 `npm run build`도 통과했다. 통합 테스트에서는 동일한
사용자 날짜가 결과 저장, 메인 달력 표시, 조언 조회에 연속해서 사용되는지
확인한다.

### 수정 4 — 프로필 없음과 조회 오류 구분

**수정일** 2026-07-24

**파일** `src/stores/authStore.js`

기존 Vue 원본과 React 이관 코드는 프로필 조회에 실패하면 오류 종류와 관계없이
`profile=null`, `profileLoaded=true`로 처리했다. 라우터는 이 조합을 신규
사용자로 판단하므로 네트워크 장애, RLS 권한 오류 또는 일시적인 Supabase
오류가 발생한 기존 사용자도 `/join/1`로 이동할 수 있었다.

다음과 같이 수정했다.

- 프로필 조회를 `.single()`에서 `.maybeSingle()`로 변경했다.
- 조회 성공 후 행이 없을 때만 `profile=null`, `profileLoaded=true`로 두어
  실제 신규 사용자로 판단한다.
- 서버 조회 오류는 삼키지 않고 호출부로 전달하며 `profileLoaded=false`를
  유지해 신규 사용자 판정에서 제외한다.
- 인증 상태 리스너와 OAuth 보조 경로에서 프로필 조회가 실패해도 유효한 로그인
  사용자를 제거하거나 가입 완료 상태로 바꾸지 않는다.
- 정상 프로필이 이미 있으면 일시적인 재조회 실패 시 기존 프로필을 지우지 않는다.

통합 테스트에서는 기존 사용자와 신규 사용자를 각각 로그인해 정상 목적지로
이동하는지 확인한다. 기존 사용자 세션에서 네트워크를 끊거나 프로필 조회를
실패시켰을 때 `/join/1`로 잘못 이동하지 않는지 확인한 뒤 다시 온라인으로
전환해 프로필 재조회가 가능한지도 검사한다.

로컬 `npm run lint:check`와 `npm run build`는 통과했다. 빌드에서 출력되는
`face-api.js`, 초기 청크 크기와 PWA deprecation 경고는 이번 변경과 무관하며
아래 별도 안정화 항목에서 계속 추적한다.

## 결함 목록

### 1. 방 비밀번호가 외부에서 읽힐 수 있음 `Critical`

**파일** `src/stores/exchange.js` 265번째 줄

**무슨 문제인가**  
교환일기 방에 비밀번호를 걸어도 실제로는 보호가 되지 않는 상태입니다.  
현재 코드는 방 비밀번호를 서버가 아닌 브라우저에서 확인합니다. 로그인한 사용자라면 누구든 데이터베이스를 직접 조회해서 모든 방의 비밀번호를 평문으로 볼 수 있습니다.

**어떻게 고치나**  
비밀번호 확인을 Supabase Edge Function(서버) 안에서 처리하도록 변경해야 합니다. 현재 `accept-exchange-invitation` 함수처럼 서버가 검증하는 방식으로 `joinRoom`을 교체하면 됩니다.

**오픈 전 필수 수정 항목입니다.**

---

### 2. 교환일기 작성 중 앱을 나가면 오늘 일기가 사라짐 `High`

**파일** `src/views/chat/ResultView.vue` 190번째 줄

**무슨 문제인가**  
감정 분석 결과 화면에서 '교환일기 작성하기'를 선택하면 오늘 일기가 저장되지 않은 채 채팅 데이터가 삭제됩니다. 교환일기 작성 화면으로 이동한 뒤 뒤로가기를 하거나 앱을 종료하면 오늘 감정 일기를 복구할 방법이 없습니다.

**어떻게 고치나**  
교환일기 작성 화면으로 이동하기 전에 오늘 일기를 먼저 저장하도록 순서를 변경합니다. 저장 완료 후에 교환일기 작성으로 넘어가면 됩니다.

---

### 3. 저장 중 네트워크가 느리면 일기가 중복 생성될 수 있음 `Medium`

**파일** `src/stores/diary.js` 79번째 줄

**무슨 문제인가**  
일기 저장 시 10초 안에 응답이 없으면 "저장 시간이 초과됐어요" 에러가 표시됩니다. 그런데 에러가 표시된 뒤에도 실제 저장 요청은 계속 진행됩니다. 11~12초 후 저장이 완료되면 DB에는 일기가 생성된 상태인데 사용자는 에러를 보고 다시 저장 버튼을 누르게 됩니다. 결과적으로 같은 날 일기가 두 개 생성됩니다.

**어떻게 고치나**  
타임아웃이 발생할 때 Supabase 요청도 함께 취소(`AbortController`)하거나, 저장 전에 같은 날짜 기록이 이미 있는지 확인하는 로직을 추가합니다.

---

### 4. 교환일기 방 생성 실패 시 목록에 빈 항목이 추가됨 `Medium`

**파일** `src/stores/exchange.js` 191번째 줄

**무슨 문제인가**  
교환일기 방을 만들 때 서버 응답 형식이 예상과 다를 경우(네트워크 오류, 서버 배포 문제 등), ID가 없는 빈 게시글 항목이 목록 화면에 추가됩니다. 이 항목은 공유하거나 삭제할 수 없고 화면에서 사라지지도 않아 혼란을 줍니다.

**어떻게 고치나**  
서버 응답에서 `room` 데이터가 정상적으로 포함되어 있는지 먼저 확인한 뒤 목록에 추가하도록 코드를 수정합니다.

---

### 5. 알림 클릭 시 의도하지 않은 화면으로 이동될 수 있음 `Medium`

**파일** `src/router/index.js` 91번째 줄

**무슨 문제인가**  
푸시 알림에 포함된 이동 경로(`notificationTarget`)를 아무 검증 없이 그대로 사용합니다. 누군가 조작된 링크를 만들면 앱 안의 임의 화면으로 이동시킬 수 있습니다.

**어떻게 고치나**  
허용된 경로 목록을 정의하고, `notificationTarget` 값이 그 안에 있는 경우에만 이동하도록 처리합니다.

### 6. 메인 화면 진입 시 달력 데이터를 두 번 불러옴 `Low`

**파일** `src/views/main/MainView.vue` 117번째 줄

**무슨 문제인가**

**어떻게 고치나**
